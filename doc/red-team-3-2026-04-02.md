## CONFIRMED FINDINGS — Sorted by Severity

### SEVERITY: HIGH (10)

#### F1. `flat/format-form` hangs on infinite lazy sequences — no print-length bound
**Source:** REPL H-PRINT-INFINITE
**Location:** `printer.cljc` (`to-doc`), `flat.cljc`, `canon.cljc`

The printer's `to-doc` realizes the entire input sequence. `(flat/format-form (range))` hangs the JVM with unbounded memory growth until OOM. Crashed the nREPL process during testing.

**Impact:** DoS in any code path passing user-provided Clojure forms to the printer.
**Fix:** Check `*print-length*` or add internal element cap in `to-doc` before realizing sequences.

#### F2. Plain integers > Long.MAX_VALUE error instead of promoting to BigInt
**Source:** Agent 2 (H-VR-22)
**Location:** `resolve.cljc:236-237`

```
Meme:    9999999999999999999  → error "Invalid number"
Clojure: 9999999999999999999  → 9999999999999999999N (BigInt)
```

The `:else` branch uses `Long/parseLong` without BigInt fallback. Hex/octal/radix already use `BigInteger` and promote correctly — only plain integers are missing the promotion.

**Fix:** `try (Long/parseLong raw) (catch NumberFormatException _ (BigInt/fromBigInteger (BigInteger. raw)))`.

#### F3. `expand-sq` missing `with-meta` for metadata on non-symbol collections inside syntax-quote
**Source:** Agent 3 (H-SQ-18)
**Location:** `expander.cljc:82-136`

In Clojure, `` `^:foo [1 2] `` expands to `(with-meta (apply vector ...) {:foo true})`. Meme's `expand-sq` handles vectors, maps, and sets but never inspects `(meta form)` and never emits `with-meta` wrapping. Metadata on collections inside syntax-quote is silently lost.

**Impact:** Macro authors writing `` `^:foo [~@args] `` silently lose the `:foo` annotation.

#### F4. `#?@` splice result leaks through prefix operators
**Source:** REPL + Agent 1 (H-RC-1, H-RC-2)
**Location:** `reader.cljc` — `parse-simple-prefix` (~line 358)

`parse-simple-prefix` only checks `discard-sentinel?`, not `splice-result?`. When `#?@(:clj [a b])` follows a prefix operator (`'`, `@`, `#'`, `` ` ``), the splice-result vector with `{::splice true}` metadata passes through unchecked.

- `'#?@(:clj [1 2])` → `(quote [1 2])` with splice metadata — should error
- `@#?@(:clj [a b])` → `(deref [a b])` with leaking metadata

#### F5. `.charAt` returns JS string not char on CLJS — char literals produce wrong type
**Source:** Agent 6 (H-XP-4, H-XP-5)
**Location:** `resolve.cljc:52,56,83`

`(.charAt inner i)` has no `#?` guard. On CLJS returns a single-char JS string, not a char. This means `MemeRaw.value` for char literals holds a JS string `"a"` on CLJS instead of char `\a` on JVM. The printer's `char?` branch is JVM-only, so on CLJS chars fall through to `pr-str` → printed as `"a"` (quoted string), not `\a`.

#### F6. Large integer precision loss via `js/parseInt` on CLJS
**Source:** Agent 6 (H-XP-8)
**Location:** `resolve.cljc:237`

`js/parseInt "9007199254740993" 10` returns `9007199254740992` (off by 1, beyond `Number.MAX_SAFE_INTEGER`). Silent precision loss with no warning.

#### F7. `meme/tagged` silently drops tag on CLJS in `transform-structures`
**Source:** Agent 6 (H-XP-26)
**Location:** `rewrite/rules.cljc:101-103`

On CLJS, `#foo [1 2 3]` is reduced to just `[1 2 3]` — the tag is silently discarded. Affects `clj→meme→clj` roundtrips on CLJS.

#### F8. `meme/reader-cond` no-match leaks ReaderConditional on rewrite path
**Source:** Agent 6 (H-XP-27)
**Location:** `rewrite/rules.cljc:105-111`

On the rewrite/TRS path, unmatched reader conditional branches are wrapped in a new `ReaderConditional` object instead of being discarded. Classic parser correctly returns `discard-sentinel` for no-match. Behavioral divergence between langs.

#### F9. Comments on primitive map keys/values silently dropped (non-IMeta)
**Source:** Agent 4 (H-PF-6, H-PF-7)
**Location:** `printer.cljc:22-29`

`form-comments` checks `(instance? IMeta form)`. Keywords, numbers, strings are NOT `IMeta`. Comment metadata attached to these forms is silently lost. In `{; comment\n :a 1}`, the comment on keyword `:a` disappears.

#### F10. Java regex syntax accepted by meme but fails or diverges in JS
**Source:** Agent 6 (H-XP-11)
**Location:** `resolve.cljc:248-256`

Java-specific patterns (possessive quantifiers, named groups with `?P<>`, variable-length lookbehind) compile on JVM but produce raw JS `SyntaxError` on CLJS, bypassing `errors/meme-error`.

## Key Themes

1. **Printer unbounded realization** (F1): Single most impactful fix — add `*print-length*` awareness
2. **Number resolution gaps** (F2, F6, F12): BigInt promotion missing for plain integers; CLJS precision loss; ratio type divergence
3. **Syntax-quote metadata loss** (F3): `expand-sq` doesn't emit `with-meta` for annotated collections — affects macro authors
4. **#?@ splice result not tracked** (F4, F15, F16): splice metadata leaks through prefix operators and into maps/sets
5. **CLJS type divergence** (F5, F7, F8, F10): char type, tagged literal tag drop, RC no-match leak, regex engine differences
6. **Stage contract gaps** (F18-F20, F28-F29): prelude bypasses validation, rewrite has no spec, options clobbered
7. **Comment preservation** (F9, F27): non-IMeta forms can't carry comment metadata; formatter repositions end-of-line comments
